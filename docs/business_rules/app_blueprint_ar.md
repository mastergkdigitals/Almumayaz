# المخطط المعتمد لتطبيق المميز ERP

> هذه الوثيقة هي المرجع الأساسي للمنتج، ويمكن تعديلها بإصدارات موثقة عند اكتشاف متطلبات جديدة أثناء تصميم الشاشات والتنفيذ.

## 1. نطاق المنتج

- نظام ERP تجاري، وليس نظام محاسبة بالقيد المزدوج.
- يجب أن تسمح البنية بإضافة المحاسبة الكاملة مستقبلاً دون إعادة تصميم النظام بالكامل.
- لا توجد ضرائب أو ضريبة قيمة مضافة (VAT) في الإصدار الأول.
- الإصدار الأول يبدأ بقاعدة بيانات جديدة ونظيفة.

### وحدات الإصدار الأول

- المستخدمون والصلاحيات
- سجل النشاط والتدقيق
- الزبائن والمجهزون
- المجموعات والأنواع والمواد
- المخازن والنقل المخزني
- المشتريات ومرتجعاتها
- المبيعات ومرتجعاتها
- الأقساط
- الصندوق
- سندات القبض والصرف
- المصاريف
- التقارير وكشوف الحساب
- الإعدادات
- النسخ الاحتياطي والاستعادة
- PDF وExcel والطباعة

لا يشمل الإصدار الأول الباركود أو الأرشيف الإلكتروني.

## 2. التقنية والتشغيل

- Flutter لتطبيق Windows المكتبي.
- Express مع TypeScript للخادم.
- PostgreSQL لقاعدة البيانات.
- اللغة العربية واتجاه RTL وخط Tajawal.
- PDF مبني من بيانات المستند، وليس صورة من الشاشة.
- تصدير Excel باستخدام مكتبة مناسبة ومعتمدة للمشروع.
- يعمل عبر LAN/Wi-Fi لعدة مستخدمين، ويدعم خادماً مخصصاً.
- يعمل النظام بالكامل دون إنترنت.
- يحتاج الإنترنت فقط لرفع النسخ الاحتياطية إلى Google Drive.
- فشل Google Drive لا يمنع العمل المحلي.
- تتصل تطبيقات المستخدمين بقاعدة البيانات من خلال API فقط.
- يمنع وضع ملفات قاعدة البيانات في مجلد شبكة مشترك.

## 3. Windows والأجهزة

- Windows Desktop فقط.
- Windows 10 كحد أدنى، مع دعم Windows 11 والإصدارات اللاحقة.
- الحد الأدنى المستهدف: Intel Core i5 الجيل التاسع أو ما يعادله، وذاكرة 8 GB.
- دقة 1280×720 هي الحد الأدنى.
- يفتح التطبيق بحالة Maximized، وليس وضع Full Screen الإجباري.
- عرض التصميم المرجعي 1440 logical pixels.
- الواجهة متجاوبة مع دقات وأحجام شاشات Windows المختلفة دون تحديد حد أعلى ثابت.
- يجب احترام Windows DPI عند 100% و125% و150% وعدم إجبار تكبير النص على 1.0.

## 4. الواجهة وتجربة الاستخدام

- تصميم مكتبي جديد ونظيف.
- نستفيد من حقول ووظائف التطبيق القديم، ولا ننسخ بنيته أو شاشاته المكررة وغير المكتملة.
- نظام مركزي واحد للتجاوب، ونقطة Scaling واحدة عند غلاف التطبيق.
- تخطيط مكتبي عادي وتخطيط مكتبي Compact عند الحاجة؛ لا توجد واجهة هاتف أو جهاز لوحي.
- الارتفاع يعالج بالقيود والتمرير، وليس بتصغير الشاشة بالكامل.
- نظام تصميم مركزي للألوان والخطوط والمسافات والحواف والظلال والارتفاعات والأيقونات والحركات والحوارات والجداول والحالات.
- شاشة DesignSystemGalleryScreen لعرض المكونات المشتركة وحالاتها.
- تجنب التأثيرات الثقيلة، والتحريك الزخرفي المستمر، وعمليات الشبكة أو الملفات أو الحسابات الثقيلة داخل build.
- الجداول الكبيرة تستخدم pagination أو lazy rendering.
- لا يدعى تحسين الأجهزة الضعيفة قبل القياس في Profile وRelease.

## 5. معايير لوحة المفاتيح

- Enter وTab: الانتقال إلى الحقل المنطقي التالي.
- Shift+Tab: الانتقال إلى الحقل السابق.
- Escape: إغلاق القائمة أو الحوار أولاً ثم الرجوع الآمن.
- Ctrl+S: حفظ.
- Ctrl+N: إنشاء مستند جديد.
- Ctrl+F: البحث.
- F5: تحديث الشاشة.
- إدارة التركيز مركزية وقابلة للضبط، وليست requestFocus موزعة داخل الشاشات.

## 6. الشركة والمخازن

- شركة واحدة لكل تثبيت.
- فرع شركة واحد.
- مخازن متعددة.
- فرع جهة عمل الشخص لا يعني فرع الشركة.
- تسلسل عالمي مستقل لكل نوع مستند.
- يولد الخادم أرقام المستندات داخل المعاملة.
- يبدأ الترقيم من 1، ولا يعاد استخدام الأرقام المحذوفة.

## 7. الأشخاص

المسميات الرسمية في الواجهة والوثائق هي:

- زبون / زبائن
- مجهز / مجهزين
- زبون ومجهز
- موظف

يمنع استخدام مصطلحي عميل أو مورد.

- نوع الشخص للتصنيف والتقارير فقط، ولا يغير الصلاحيات أو القواعد الحسابية.
- الموظف للتصنيف والتقارير فقط؛ لا يوجد نظام رواتب في الإصدار الأول.
- البيانات تشمل الاسم، الهاتف، الهاتف الإضافي، جهة العمل، فرع جهة العمل، المدينة، العنوان والملاحظات.
- الرصيد الافتتاحي منفصل للدينار والدولار.
- كشوف الحساب تدعم البحث والفترة والعملة والطباعة وPDF وExcel.

## 8. العملات والقيم

- يدعم التطبيق الدينار العراقي والدولار الأمريكي فقط، ولا توجد عملات إضافية.
- العملة الأساسية هي IQD.
- يحفظ سعر الصرف داخل كل فاتورة شراء أو بيع.
- تعرض مبالغ IQD كدنانير صحيحة.
- تعرض مبالغ USD بمنزلتين عشريتين.
- تخزن القيم المالية وأسعار الصرف داخلياً بدقة أعلى باستخدام NUMERIC.
- ترسل القيم العشرية عبر API كنصوص لتجنب أخطاء floating point.
- الكميات أعداد صحيحة فقط مثل 1 أو 100 أو 1000؛ لا يسمح بالكميات الكسرية.

## 9. المواد والمخزون

- رمز المادة يدخله المستخدم ويجب أن يكون فريداً.
- لا يوجد باركود في الإصدار الأول.
- للمادة وحدة واحدة في الإصدار الأول.
- يمكن إضافة مادة بسرعة من داخل الفاتورة بصلاحية مستقلة.
- المتوسط المتحرك للتكلفة يحسب لكل مخزن.
- المخزون يتأثر فور حفظ المشتريات والمبيعات والمرتجعات والنقل.
- المخزون السالب ممنوع افتراضياً.
- يمكن السماح به من الإعدادات و/أو بصلاحية حساسة.
- جميع العمليات متعددة الجداول ذرية وتستخدم transactions وrow locking عند الحاجة.

## 10. الفواتير والمرتجعات

- التسمية الرسمية: فاتورة شراء وفاتورة بيع.
- لا توجد حالة Draft/Posted؛ الحفظ يطبق الأثر فوراً.
- الدفع يمكن أن يكون كاملاً أو جزئياً أو صفراً.
- خصم على السطر وخصم على الفاتورة.
- يمكن إدخال الخصم كنسبة أو مبلغ، ويحسب النظام القيمة الأخرى.
- اسم السائق في فاتورة البيع حقل كتابة حرة.
- توجد طباعة عادية وطباعة بدون أسعار للمشتريات والمبيعات.
- مرتجع البيع أو الشراء مستند مستقل ومرقم ومرتبط بالفاتورة الأصلية.
- المستخدم المخول يستطيع تعديل الفواتير وحذفها نهائياً.
- التعديل يعكس الأثر السابق ويطبق الأثر الجديد داخل معاملة واحدة.
- الحذف لا يمكن استعادته، لكن يحتفظ التدقيق بمعلومات كافية لمعرفة المستند المحذوف وآثاره السابقة.
- لا يعاد استخدام رقم المستند المحذوف.

## 11. النقل المخزني

- النقل يؤثر في المخزون فور الحفظ.
- لا يمكن تعديل أو حذف النقل المحفوظ.
- يصحح الخطأ بحركة عكسية مستقلة ومسجلة.
- يجب التحقق من المخازن والكميات والرصيد ومنع النقل إلى المخزن نفسه.

## 12. الأقساط

- الأقساط جزء من الإصدار الأول.
- لا توجد فوائد أو زيادة.
- ترتبط خطة الأقساط بفاتورة البيع.
- تتضمن المبلغ وتاريخ الاستحقاق وحالة التسديد والملاحظات.
- يؤثر تسديد القسط في رصيد الزبون والصندوق داخل معاملة واحدة.

## 13. الصندوق والسندات والمصاريف

- صندوق رئيسي واحد برصيد منفصل للدينار والدولار.
- حسابات رئيسية وفرعية قابلة للتخصيص.
- حسابات النظام محمية من التعديل أو الحذف غير الآمن.
- سند القبض أو الصرف يمكن أن يستخدم IQD فقط أو USD فقط أو العملتين معاً.
- يجب أن يحتوي السند على مبلغ موجب واحد على الأقل.
- المصروف يمكن أن يكون مدفوعاً فوراً أو غير مدفوع ومستحقاً لمجهز.

## 14. المستخدمون والصلاحيات

- حساب Administrator أول بصلاحية كاملة.
- لا تخزن كلمة مرور افتراضية ثابتة داخل الكود.
- عدد المستخدمين غير محدود.
- يمكن توفير قوالب أدوار اختيارية مع صلاحيات قابلة للتعديل.
- الصلاحيات حسب الشاشة والفعل: عرض، إنشاء، تعديل، حذف، طباعة، تصدير، استعادة نسخة، تغيير الإعدادات والعمليات الحساسة.
- يفرض API المصادقة والصلاحيات؛ إخفاء زر في Flutter ليس حماية كافية.
- تستخدم كلمات مرور مشفرة بخوارزمية قوية.

- مجلد محلي يحدده المستخدم، مع Google Drive اختيارياً.
- ملفات النسخ مشفرة ولا تستعاد إلا من التطبيق.
- الاستعادة للمدير فقط.
- فحص سلامة النسخة وتوافقها قبل الاستعادة.

## 18. البنية البرمجية

### Flutter

الاتجاه الأساسي:

Widget → Controller → Feature Repository/API Gateway → Shared HTTP Transport → Backend

- بنية feature-first.
- الشاشات تركب الواجهة ولا تنشئ طلبات HTTP أو تحتوي قواعد العمل.
- نقل الحسابات والتحقق والسياسات إلى طبقات واضحة.
- اتصال HTTP مشترك ودائم، مع API gateways خاصة بالميزات.
- Dependency injection خفيف.
- تقسيم الملفات حسب المسؤولية، ومراجعة أي ملف يتجاوز 400 سطر، وتقسيم الملفات الكبيرة دون أجزاء part1/part2.

### Backend

للوحدات المعقدة:

routes → controller → schema/validation → service → repository → PostgreSQL

- routes للمسارات والـmiddleware.
- controller لترجمة HTTP.
- schema للتحقق والتطبيع.
- service لقواعد العمل وتنسيق المعاملات.
- repository لـSQL والقفل والحفظ.
- mapper للتحويل بين صفوف قاعدة البيانات ونماذج API.
- مصادقة وصلاحيات وأخطاء وlogging وrequest IDs وenvironment validation وtransaction helpers وpagination مشتركة.

## 19. API وقاعدة البيانات

- جميع المسارات تحت /api/v1.
- شكل نجاح موحد يحتوي data وmeta.requestId.
- شكل خطأ موحد يحتوي code ثابتاً ورسالة عربية آمنة وdetails وrequestId.
- لا تعرض stack traces للمستخدم.
- Pagination وفلترة وترتيب صريح للقوائم المتنامية.
- التواريخ التجارية YYYY-MM-DD، والطوابع الزمنية UTC ISO.
- النص الاختياري الفارغ يطبع إلى null بصورة موحدة.
- حماية التزامن باستخدام version أو updatedAt.
- PostgreSQL مع UUID للكيانات.
- مفاتيح أساسية وخارجية وunique وcheck constraints وفهارس حسب الاستعلامات.
- migrations إضافية وغير قابلة للتغيير بعد النشر، مع checksums.
- seeds مرجعية idempotent.
- اختبار migrations على قاعدة فارغة وقاعدة موجودة.
